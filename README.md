# 水果打包加工 Demo

一个手机版最小可运行闭环：录入原料果 A/B、成品包数、次果和损耗，计算库存变化与成本分摊。

## 1. 怎么启动、怎么跑、怎么验证

### 启动

方式一：直接用浏览器打开 `https://xuyidan0919.github.io/Fruit-processing-and-packaging/`。

### 怎么跑

1. 进入录入页。
2. 可以直接使用默认数据，也可以点击“示例”恢复示例数据。
3. 点击“生成结果”。
4. 如果数据合法，会进入结果页；如果数据不合法，会停留在录入页并提示错误。


## 2. 设计说明

### 为什么这么设计

这个题目的重点是可运行、可演示的最小闭环，不是完整业务系统。所以当前实现选择纯前端静态页面：

- 不依赖后端、数据库、构建工具，交付和打开成本低。
- 业务链路集中在一次加工批次：录入 -> 校验 -> 计算 -> 展示。
- 先把库存守恒、次果、损耗、成本结转讲清楚，避免堆功能。

### 核心结构

```text
index.html   页面结构：录入页、结果页
styles.css   手机优先样式
app.js       表单读取、校验、库存计算、成本计算、页面切换
README.md    交付说明
```

### 核心规则

- 原料果 A 成本价：`2 元/斤`
- 原料果 B 成本价：`3 元/斤`
- 成品规格限制为：`5 斤装`、`8 斤装`、`10 斤装`
- 成品重量 = 规格斤数 × 包数
- 产出重量 = 成品重量 + 次果 + 损耗
- 守恒差异 = 原料投入 - 产出重量
- 成本按成品、次果、损耗的重量占比分摊并分别列示

### 已验证

## 4. 风险与限制

### 已知风险

- 成本分摊规则是当前实现假设：按成品、次果、损耗重量占比分摊。题目没有给唯一成本处理方式。
- 不守恒数据被设置为阻断错误。这样更稳，但如果业务允许暂存异常加工单，需要后续调整。

### 当前仍可接受的原因

- 目标是最小可运行 demo，不是完整生产系统。
- 已覆盖核心闭环：原料投入、成品产出、次果、损耗、库存守恒、成本结转。
- 关键不合理输入会被阻断，避免展示误导性结果。

## 5. 我对业务场景、库存和成本处理的理解

- 这个场景是一批原料果进入加工环节后，被拆分成多类产出的加工入库问题。
- 库存上，原料果 A、B 被消耗，库存减少；加工后形成不同规格的成品果，成品库存增加；加工过程中产生次果和损耗，也需要被记录；整个过程要满足重量守恒：原料 = 成品 + 次果 + 损耗。
- 成本上，我的处理思路是，先按原料投入计算总成本，再把这批原料总成本结算到加工结果中。
因为题目没有给出更细的成本分配规则，我采用按重量占比分摊的方式，将成本分摊到成品、次果和损耗上，并分别展示，这样能保证成本总额不丢失，也能让每类产出都有可解释的成本来源。

## 6. 我主动补充的规则和假设

- 成品规格固定为 5 斤装、8 斤装、10 斤装
- 成品重量 = 规格斤数 × 包数
- 次果和损耗都按“斤”录入
- 投入产出必须守恒，不守恒时阻断生成结果
- 成本按成品、次果、损耗的重量占比分摊
- 当前只处理单个加工批次，不做多批次历史
- 当前不计算售价、收入和利润
- 不追踪原料 A/B 分别流向哪个成品规格

这些假设的目的让最小 demo 能形成闭环，并避免出现库存或成本解释不清的问题。
